Succesful code start with the right style.

When people are working on a project, most of them just write code based on an idea that just comes to mind.
They do not comment their written code, nor are they consistent on the style how they write different statements.
This styling guide is a reference for C programming in embedded systems, but parts may be used in other languages or for other purposes.

Curly bracket

Curly brackets {} should be placed on a seperate line. This means that after ANY statement (if, for, while etc) a newline should be added containing the curly bracket.
The reason for doing is, is that missing curly brackets could be more easily detected, because they ALWAYS got the same amount of tabs before the curly bracket. A programmer should look vertical to see if an opening curly bracket meets a vertical curly bracket and vice versa.


Let me explain the missing curly bracket by looking vertical in the code.
If we look at the correct way of writing the curly bracket on the next line (see the code on the right), we can see that both curly brackets contains 1 tab space before the actual curly bracket is written. They are in line with each other.
If we would write a bigger statement and write the curly bracket on the same line, the curly bracket may not be even onto the screen. This causes that the programmer need to scroll to look at the last character of the statement and verify if this is a curly bracket or not.

Writing the curly bracket on a new line in line with the other (the same amount of tabs) it is way easier to find out if there is a mission curly bracket or not. 

						
	int mem_copied = 1;	/* variables always in snake_case */
	if(mem_copied) {	/* BAD! The curly bracket should be on next line */
		mem_copied = 0;	
	}

						
					
						
	int i;				
	for(i = 0; i < otherval; i++)	/* Curly bracket on a seperate line */
	{				/* A missing curly bracket is easier to detect */
		i += 5;
	}

Structs and unions 

						

	typedef struct		/* Typedef not recommended, but allowed */ 
	{
		int an_int;	/* Struct members in snake_case */
		char a_char;
	} object_name_t;	/* Object name in snake_case */
				/* Because typedef, add _t at the end */

							
					
						
	union union_t	/* Prefered method, no typedef */
	{
		int i;	/* Union member in snake_case */
		char c;
	}

A struct and a union may be typedeffed, but this is not recommended. If you typedef a struct or a union, you are actually hiding some important information for other programmers. It is legal to use, but do not typedef if you are wisely.
Struct and union members should be written in snake_case, since they are not constants. Writing them in another style, like CamelCase is strictly forbidden and is not allowed.


As you can see in the code, the curly brackets are on new lines as discussed before. Keep in mind that structs and unions should be declared as much as possible in an header file (.h extension). More on this later.

Enums

Enums are a slightly bit different than structs and unions. Since enums are integers that start from 0, they can be seen as constants. Constants in this sentence means that enum elements should be all capitaliezd.
After each enum element there should be a comma next to the last character of the enum element. NO spaces are allowed.

If needed, you may change the default value of an element by using the assign operator followed by a space and the value. The rule is the same, the comma is placed after the last number and no spaces are between the comma and the last digit. 

						
	enum 
	{
		FALSE ,		/* BAD. No space between last character and comma */
		TRUE
	}

	enum
	{
		SUCCESS,	/* Every element contains a comma after the last character */
		ERROR,
		WARNING,
		EXIT		/* The last element does NOT contain a comma */
	}

Conditions 

						
	if(value != 0)	/* BAD. This result in longer lines and is redundant */
	if(value)	/* Check if the value is not equal to 0 */				
						
					
						
	if(value == 0)	/* BAD. We want our code to be as small as possible */
	if(!value)	/* Check if the value is equal to 0 */
						
					
						
	if(value=>5)	/* BAD. This looks familiar to the arrow operator */
	if(value => 5)	/* The correct way to write a condition */

Conditions are used in an if-, while-, for-, and switch statement. If a programmer want to compare a value that may not be equal to 0, the programmer should not write != 0. Any statement is true if the value is not equal to 0. The progammer should only write his value inside the statement.



If a programmer want to verify that a variable is equal to 0, he should not write == 0. Instead we use the logic NOT operator. Statements are true if a variable is not equal to zero. If we add the logic NOT operator (!) a statement would run true if the value is equal to zero.



It is important to improve the readability of the source code for other human programmers. To comply with this, the space bar is important in conditional statements.
A space should be added between a variable and an operator (and vice versa).

Preprocessor defines

It's standard to define preprocessor constant with all capital letters. To seperate 2 words from eachother, use an underscore _.
Macro names should be all capitalized, since they are made with the #define keyword of the preprocessor.
I have been thinking if the definition of the macro arguments should be capatilized aswell, but could not give a valid reason why it may not.

since GNU uses capitalized letters for macro names it is the best to follow this. I know that GNU does define its macro's with small letters, and I do not agree with them on this point. Please use capitalized letters on #defines all the time.

Do not forget alignment. After the #define keyword should be a tab, which uses only 1 space since a tab is 4 spaces and #define is 7 characters. Next, write your constant or macro name followed by a tab or two so the definition is aligned with each other. 

						
	#define	PI	3.14159265359
	#define	MULT(X)	X*X
						
					
						
	#define	PI	3.14159265359		/* BAD. PI should have another tab */
	#define	EULER_CONST	2.71828	

	#define	PI		3.14159265359	/* Definition of PI and EULER are aligned */
	#define	EULER_CONST	2.71828
